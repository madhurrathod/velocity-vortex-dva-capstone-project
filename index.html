Program Overview

Capstone 2 is an industry-style data analytics simulation where teams work end to end: raw dataset sourcing, Python ETL, exploratory and statistical analysis, Tableau dashboarding, and business recommendations that lead to decisions rather than just charts.

What matters most

This capstone rewards analytical thinking, reproducibility, and decision-oriented storytelling. Tools support the work, but marks come from problem framing, sound methodology, and practical recommendations backed by evidence.

Overview

Each team selects a sector and business problem, builds the Python workflow, prepares an analysis-ready dataset, develops a Tableau Public dashboard, and submits a final report, presentation, and GitHub repository with visible contribution history.

Team Structure

  • Team size is fixed at 5 members.
  • Teams are student-formed for balanced work allocation.
  • No swapping is permitted after formation.

Primary Objective

  • Demonstrate full ETL capability using Python.
  • Show strong EDA and statistical depth.
  • Deliver insights and recommendations for decisions.

Core Tools

  • Python for extraction, cleaning, analysis, and KPI prep.
  • Tableau Public for interactive dashboards and storytelling.
  • GitHub for version control, PR workflow, and contribution audit.

Execution Schedule

The brief describes a 10-day working schedule across two weeks, while the orientation page presents the overall capstone window from April 13, 2026 to April 28, 2026. Treat the day-by-day plan as the operational checklist.

Days 1-2

Discovery and Setup

Meet the team, assign roles, choose the sector, identify the problem statement, create the GitHub repository, and prepare the required folder structure.

Day 3

Gate 1 Approval

Submit the problem statement, dataset link, and initial data dictionary. Do not proceed to deep analysis until the mentor approves the primary dataset.

Days 4-5

ETL and Cleaning

Commit the raw file into data/raw/, build the cleaning flow in notebooks/02_cleaning.ipynb, and save the cleaned output in data/processed/.

Days 6-7

EDA and Statistical Analysis

Run EDA in 03_eda.ipynb, statistical work in 04_statistical_analysis.ipynb, and KPI prep in 05_final_load_prep.ipynb.

Days 8-9

Dashboard Build

Design a Tableau Public dashboard with interactive filters that answers the chosen business problem directly.

Day 10

Final Submission

Finalize the report, deck, recommendations, dashboard links, screenshots, and all GitHub assets before the deadline.

Project Workflow

The work should read like one coherent story across the repo, notebooks, dashboard, and report.

1. Define the Problem

Select the sector, define the business question, and make sure the scope can lead to measurable KPIs and clear recommendations.

2. Prepare the Data

Load, inspect, clean, standardize, and document every transformation in Python. Never overwrite the original raw file.

3. Analyze the Data

Go beyond counts and averages. Use trends, distributions, outlier checks, correlation, regression, hypothesis tests, or segmentation when relevant.

4. Build the Dashboard

Use Tableau Public for an interactive, decision-focused dashboard. Avoid decorative dashboards that do not answer the core question.

5. Write Recommendations

Turn findings into practical actions with business impact, limitations, and future scope instead of just chart descriptions.

6. Submit and Defend

Package the repo, report, screenshots, dashboard link, and presentation so every team member can explain the data, methodology, KPIs, and recommendations in viva.

Dataset Rules

The chosen data must be raw enough to require meaningful cleaning and robust enough to support analysis, KPI design, and Tableau visualization.

Minimum Standards

  • At least 8 meaningful analytical columns.
  • Tabular, row-level records.
  • Missing values, inconsistent formats, or category issues should be present.
  • No pre-cleaned or competition-ready files.
  • Prepare one primary dataset and two backups from the same sector.

Kaggle Rules

  • Use raw or minimally processed datasets only.
  • Avoid files with pre-built dashboards or engineered features.
  • Prefer searches like raw, original, logs, records, or events.
  • Mentors may reject datasets that are already analysis-ready.

Approved Dataset Sources

  • Google Dataset Search
  • Kaggle Datasets
  • Data.gov.in
  • Harvard Dataverse
  • World Bank Microdata
  • UCI Machine Learning Repository
  • data.world
  • DataHub

Submission Checklist

Every final asset should support the same narrative: problem, cleaned data, analysis, dashboard, recommendations, and proof of contribution.

GitHub Repository

  • Public repo with the correct naming convention.
  • All notebooks committed in .ipynb format.
  • data/raw/ contains the original dataset.
  • data/processed/ contains the cleaned output.
  • docs/data_dictionary.md is complete.
  • README.md explains the project, dataset, and team.
  • All members have visible commits and PRs.

Tableau Dashboard

  • Published on Tableau Public.
  • Includes at least one interactive filter.
  • Directly addresses the business problem.
  • URL shared in tableau/dashboard_links.md.
  • Screenshots committed in tableau/screenshots/.

Report and Deck

  • Project report PDF of roughly 10 to 15 pages.
  • Executive summary, methodology, analysis, and dashboard sections.
  • 8 to 12 key insights in decision language.
  • 3 to 5 actionable business recommendations.
  • Contribution matrix aligned with GitHub history.

Evaluation Rubric
